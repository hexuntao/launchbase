LaunchBase 的主要产品界面从 `apps/web` 部署。docs app 可以按需作为独立 Vercel 项目部署，但当前验证过的生产路径是 web app。

## 准备仓库

```bash
git clone https://github.com/<your-org>/<your-repo>.git
cd <your-repo>
pnpm install
```

本地验证时复制 env 示例并启动服务：

```bash
cp .env.example .env
cp apps/web/.env.example apps/web/.env
pnpm docker:up
```

生成 Better Auth secret：

```bash
openssl rand -base64 32
```

## 部署前验证

```bash
pnpm lint
pnpm typecheck
pnpm build
```

web-only 部署检查：

```bash
pnpm --filter web typecheck
pnpm web:build
```

## Vercel 项目设置

连接仓库并为 web app 创建 Vercel project：

```txt
Framework Preset: Next.js
Root Directory: apps/web
Install Command: pnpm install --frozen-lockfile
Build Command: pnpm web:build
Output Directory: .next
Node.js Version: 22.x
```

构建命令需要通过 Turborepo 使用 workspace packages。如果你的 Vercel 设置会从 `apps/web` 内部运行命令，请把 Root Directory 调回仓库根目录，并保留 `pnpm web:build`。

## 必填环境变量

在 Vercel 的 production 和 preview environments 中设置：

- `DATABASE_URL`
- `UPSTASH_REDIS_REST_URL`
- `UPSTASH_REDIS_REST_TOKEN`
- `BETTER_AUTH_SECRET`

生产 `DATABASE_URL` 应指向托管 PostgreSQL。生产 Redis 值应指向 Upstash Redis 或兼容的 Redis REST endpoint。

## 可选集成

- Google OAuth：`GOOGLE_CLIENT_ID`、`GOOGLE_CLIENT_SECRET`。
- Asset / CSP origin：`NEXT_PUBLIC_ASSET_ORIGIN`。
- PostHog：`NEXT_PUBLIC_POSTHOG_KEY`、`NEXT_PUBLIC_POSTHOG_HOST`。
- Sentry：`NEXT_PUBLIC_SENTRY_DSN`、`NEXT_PUBLIC_SENTRY_CSP_REPORT_ENDPOINT`、`SENTRY_ORG`、`SENTRY_PROJECT`、`SENTRY_AUTH_TOKEN`。

未启用集成时保持可选值为空。不要提交真实 token 或 DSN。

## Monorepo 注意事项

- 保持 `pnpm-lock.yaml`、`pnpm-workspace.yaml` 和 `turbo.json` 位于仓库根目录。
- 不要把共享 packages 复制进 `apps/web`；workspace dependency graph 就是契约。
- 不要提交生成的 `.next` 输出。
- 如果单独部署 docs，为 `apps/docs` 创建第二个 Vercel project，并使用 `pnpm docs:build`。

## 常见错误

- `BETTER_AUTH_SECRET` 太短：生成至少 32 个字符的值。
- 数据库连接失败：确认 `DATABASE_URL` 可从 Vercel 访问，并符合数据库提供商的 SSL 要求。
- Redis env validation 失败：同时设置 `UPSTASH_REDIS_REST_URL` 和 `UPSTASH_REDIS_REST_TOKEN`。
- OAuth callback 不匹配：启用 Google 登录前，把部署域名加入 Google OAuth redirect URIs。
- 构建无法解析 workspace packages：确认 install 从 monorepo 根目录运行，并使用 pnpm。
