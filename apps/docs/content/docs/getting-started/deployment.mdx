LaunchBase deploys the primary product surface from `apps/web`. The docs app can be deployed as a separate Vercel project if needed, but the verified production path is the web app.

## Prepare the repository

```bash
git clone https://github.com/<your-org>/<your-repo>.git
cd <your-repo>
pnpm install
```

For local validation, copy env examples and start local services:

```bash
cp .env.example .env
cp apps/web/.env.example apps/web/.env
pnpm docker:up
```

Generate a Better Auth secret:

```bash
openssl rand -base64 32
```

## Validate before deployment

```bash
pnpm lint
pnpm typecheck
pnpm build
```

For a web-only deployment check:

```bash
pnpm --filter web typecheck
pnpm web:build
```

## Vercel project setup

Create a Vercel project connected to the repository and configure it for the web app:

```txt
Framework Preset: Next.js
Root Directory: apps/web
Install Command: pnpm install --frozen-lockfile
Build Command: pnpm web:build
Output Directory: .next
Node.js Version: 22.x
```

The build command runs from the repository root context through Turborepo so workspace packages remain available. If your Vercel setup runs commands from `apps/web`, switch the Root Directory back to repository root and keep `pnpm web:build`.

## Required environment variables

Set these values in Vercel for production and preview environments:

- `DATABASE_URL`
- `UPSTASH_REDIS_REST_URL`
- `UPSTASH_REDIS_REST_TOKEN`
- `BETTER_AUTH_SECRET`

Production `DATABASE_URL` must point to a managed PostgreSQL database. Production Redis values should point to Upstash Redis or another compatible Redis REST endpoint.

## Optional integrations

- Google OAuth: `GOOGLE_CLIENT_ID`, `GOOGLE_CLIENT_SECRET`.
- Asset / CSP origin: `NEXT_PUBLIC_ASSET_ORIGIN`.
- PostHog: `NEXT_PUBLIC_POSTHOG_KEY`, `NEXT_PUBLIC_POSTHOG_HOST`.
- Sentry: `NEXT_PUBLIC_SENTRY_DSN`, `NEXT_PUBLIC_SENTRY_CSP_REPORT_ENDPOINT`, `SENTRY_ORG`, `SENTRY_PROJECT`, `SENTRY_AUTH_TOKEN`.

Leave optional values empty unless the integration is enabled. Never commit real tokens or DSNs.

## Monorepo notes

- Keep `pnpm-lock.yaml`, `pnpm-workspace.yaml`, and `turbo.json` at the repository root.
- Do not copy shared packages into `apps/web`; the workspace dependency graph is the contract.
- Keep generated `.next` output uncommitted.
- If deploying docs separately, create a second Vercel project for `apps/docs` with `pnpm docs:build`.

## Common errors

- `BETTER_AUTH_SECRET` too short: generate a value with at least 32 characters.
- Database connection fails: confirm `DATABASE_URL` is reachable from Vercel and uses the correct SSL requirements for your provider.
- Redis env validation fails: set both `UPSTASH_REDIS_REST_URL` and `UPSTASH_REDIS_REST_TOKEN`.
- OAuth callback mismatch: add the deployed domain to your Google OAuth redirect URIs before enabling Google sign-in.
- Build cannot resolve workspace packages: verify install runs from the monorepo root and uses pnpm.
